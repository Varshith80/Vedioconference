# Logging

> How the platform produces, structures, transports, and retains
> logs, and what is and is **not** allowed to appear in a log line.

## 1. Goals

1. Every request has a unique **request id** that flows from the
   edge (Vercel) through the Next.js app, into n8n, and into Sentry.
2. Every server log line is **structured JSON** so it can be parsed
   by any log aggregator.
3. **No PII** (email, phone, full name, address) is logged in clear
   text. A `redact()` helper is called from every log site.
4. The audit trail is **append-only** and cannot be tampered with
   from the application layer.

## 2. Log levels

| Level   | Used for | Example |
|---------|----------|---------|
| `debug` | verbose dev information | SQL parameters, response bodies |
| `info`  | normal lifecycle events   | "booking created", "email sent" |
| `warn`  | recoverable anomalies     | "reminder duplicate skipped" |
| `error` | unexpected failures       | "Stripe 5xx after 3 retries" |

`LOG_LEVEL` controls the floor. Default: `info` in production,
`debug` in development.

## 3. Logger interface

`apps/web/lib/utils/logger.ts`:

```ts
logger.info('booking.created', { bookingId, userId, amountCents });
logger.error('stripe.session_create_failed', { userId, err: redact(err) });
```

The `redact()` helper strips:

- `email`   → first char + `***@` + last 3 chars of the domain
- `phone`   → last 4 digits
- any value of `*.password`, `*.token`, `*.secret`, `*.apiKey`
- any value whose key is `Authorization`

## 4. Request id / correlation id

- Generated by the Next.js middleware: `crypto.randomUUID()`.
- Stored in the request header `x-request-id`.
- Forwarded to n8n as `X-Request-Id` on every outbound HTTP call.
- Returned to the browser in the response header `x-request-id`.
- Included in every `logger.*` call site via the AsyncLocalStorage
  context (Phase 2 implementation).

## 5. Audit log

The `audit_logs` table is **separate** from the application logs:

- Application logs go to Sentry + Vercel logs (ephemeral).
- `audit_logs` lives in Postgres, is append-only, and is read by
  the admin dashboard.

Triggers: `bookings`, `payments`, `profiles`. New tables are added
in Phase 4 (`tutors`, `courses`, `resources`).

## 6. What we log

### 6.1 Authentication

- `auth.signup.succeeded { userId, requestId }`
- `auth.signin.succeeded { userId, requestId }`
- `auth.signin.failed { emailHash, reason, requestId }` (we hash
  the email so the log is searchable without leaking PII)
- `auth.password_reset.requested { emailHash }`
- `auth.password_reset.succeeded { userId }`

### 6.2 Bookings

- `booking.created { bookingId, userId, courseId, amountCents }`
- `booking.checkout_started { bookingId, stripeSessionId }`
- `booking.confirmed { bookingId, zoomMeetingId }`
- `booking.cancelled { bookingId, refundedCents, reason }`
- `booking.rescheduled { oldId, newId }`

### 6.3 Payments

- `payment.succeeded { paymentId, bookingId, amountCents }`
- `payment.failed { paymentId, bookingId, reason }`
- `payment.refund.issued { paymentId, amountCents }`

### 6.4 n8n

- `n8n.workflow.started { workflow, runId, requestId }`
- `n8n.workflow.completed { workflow, runId, durationMs }`
- `n8n.workflow.failed { workflow, runId, error }`
- `n8n.dead_letter { workflow, originalEvent, error }`

### 6.5 Admin actions

- `admin.user.role_changed { actorId, targetId, from, to }`
- `admin.course.deleted { actorId, courseId }`
- `admin.resource.uploaded { actorId, resourceId }`

## 7. What we **do not** log

- Passwords (plain or hashed).
- Session tokens, JWTs, refresh tokens.
- API keys / OAuth secrets.
- Credit card numbers or CVV (Stripe never sends them to us).
- Full body of any inbound or outbound HTTP request by default;
  optional `LOG_LEVEL=debug` enables it for dev only.

## 8. Retention

| Stream | Retention |
|---|---|
| Vercel runtime logs | 7 days (free), 30 days (pro) |
| Sentry events | 90 days |
| `audit_logs` (Postgres) | 7 years (financial records) |
| `notifications` (Postgres) | 2 years (in-app) |
| Stripe dashboard logs | 30 days (then Stripe Pay/Teams) |
| n8n execution logs | 14 days (configurable) |

## 9. Access control

- `audit_logs` is admin-read-only via RLS.
- Application logs are viewable by SRE and on-call.
- PII redaction is enforced at the log site, not at the destination
  (defence in depth: even if a log leaks, PII is partial).

## 10. Sensitive-data rules

1. **Never** call `JSON.stringify` on a raw request body.
2. **Always** use the `redact()` helper for any object that may
   contain PII.
3. **Always** include a request id so a leak can be correlated.
4. **Always** opt in to debug logging per environment, not globally.
