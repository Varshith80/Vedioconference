# Database

> Source of truth for the data layer. Anything that contradicts this
> document is a bug in the code.

> **Sprint B2 change — module-based workflow.** The data model is
> now two-tier: a **course** is a paid **enrollment**; a **module
> is a booked session inside that enrollment**. The legacy
> `bookings` table has been renamed to `_bookings_legacy` and is
> no longer read by any new code. See §3.1 and §4 for the new
> tables and the new lifecycle enums.

## 1. Conventions

- All primary keys are `uuid` generated by `gen_random_uuid()`.
- Timestamps are `timestamptz` (UTC). The application is responsible
  for rendering in the user's locale.
- Money is stored as **integer cents** (`integer` column).
- Soft delete is **not** used; we hard-delete where required
  (resources, audit logs are append-only).
- Every mutable table has `created_at` and `updated_at` columns with a
  `BEFORE UPDATE` trigger that bumps `updated_at`.

### 3.1 New tables in detail

#### `modules`

```
modules
  id                    uuid PK
  course_id             uuid FK → courses(id) ON DELETE CASCADE
  position              integer NOT NULL          -- 1..N, order within the course
  slug                  text NOT NULL             -- "module-1-algebre"
  title                 text NOT NULL
  description           text
  duration_min          integer NOT NULL DEFAULT 60
  is_published          boolean NOT NULL DEFAULT false
  is_preview            boolean NOT NULL DEFAULT false  -- free preview (no UI yet)
  calendly_event_uri    text                          -- per-module Calendly event type
  metadata              jsonb NOT NULL DEFAULT '{}'
  created_at, updated_at
  UNIQUE (course_id, position)
  UNIQUE (course_id, slug)
```

#### `enrollments`

```
enrollments
  id                        uuid PK
  student_id                uuid FK → profiles(id) ON DELETE RESTRICT
  course_id                 uuid FK → courses(id)  ON DELETE RESTRICT
  status                    enrollment_status NOT NULL DEFAULT 'pending_payment'
  stripe_session_id         text
  stripe_payment_intent_id  text
  amount_cents              integer NOT NULL
  currency                  char(3) NOT NULL DEFAULT 'EUR'
  paid_at                   timestamptz
  refunded_at               timestamptz
  refunded_amount_cents     integer NOT NULL DEFAULT 0
  completed_at              timestamptz
  metadata                  jsonb NOT NULL DEFAULT '{}'
  created_at, updated_at
  -- Partial unique index (see §5):
  -- CREATE UNIQUE INDEX uq_enrollments_active_student_course
  --   ON enrollments (student_id, course_id)
  --   WHERE status IN ('pending_payment', 'active');
```

A re-enrollment (`completed → active` or `refunded → active`) is
allowed because the partial index only covers `pending_payment` and
`active` rows; the old row stays in `completed` / `refunded` for
history.

#### `module_progress`

```
module_progress
  id                    uuid PK
  enrollment_id         uuid FK → enrollments(id) ON DELETE CASCADE
  module_id             uuid FK → modules(id)     ON DELETE CASCADE
  status                module_progress_status NOT NULL DEFAULT 'not_started'
  started_at            timestamptz
  completed_at          timestamptz
  created_at, updated_at
  UNIQUE (enrollment_id, module_id)
```

#### `module_bookings`

```
module_bookings
  id                    uuid PK
  enrollment_id         uuid FK → enrollments(id)  ON DELETE CASCADE
  module_id             uuid FK → modules(id)      ON DELETE RESTRICT
  tutor_id              uuid FK → tutors(id)       ON DELETE RESTRICT
  student_id            uuid FK → profiles(id)     ON DELETE RESTRICT
  status                booking_status NOT NULL DEFAULT 'scheduled'
  scheduled_start       timestamptz NOT NULL
  scheduled_end         timestamptz NOT NULL
  timezone              text NOT NULL DEFAULT 'Europe/Paris'
  calendly_event_uri    text
  calendly_invitee_uri  text UNIQUE
  notes                 text
  cancelled_at          timestamptz
  cancelled_reason      text
  rescheduled_from      uuid REFERENCES module_bookings(id) ON DELETE SET NULL
  metadata              jsonb NOT NULL DEFAULT '{}'
  created_at, updated_at
  CHECK (scheduled_end > scheduled_start)
```

`scheduled_start` / `scheduled_end` are filled by the Calendly
webhook — never by the client.

#### `payments` (changed)

```
payments
  + enrollment_id        uuid REFERENCES enrollments(id) ON DELETE CASCADE  -- preferred
  + module_booking_id    uuid REFERENCES module_bookings(id) ON DELETE CASCADE  -- future
  booking_id             uuid REFERENCES _bookings_legacy(id) ON DELETE CASCADE  -- legacy only
  ...
```

The application always writes `enrollment_id`; `booking_id` is
kept only for the legacy `_bookings_legacy` rows that may still
exist after the migration.

#### `meeting_links` (changed)

```
meeting_links
  + module_booking_id    uuid REFERENCES module_bookings(id) ON DELETE CASCADE  -- preferred
    booking_id           uuid REFERENCES _bookings_legacy(id) ON DELETE CASCADE  -- legacy only
  ...
```

`module_booking_id` is `UNIQUE`. New code never writes
`booking_id`.

#### `resource_grants` (changed)

```
resource_grants
  resource_id     uuid FK → resources(id)     ON DELETE CASCADE
  enrollment_id   uuid FK → enrollments(id)   ON DELETE CASCADE  -- replaces booking_id
  granted_at      timestamptz NOT NULL DEFAULT now()
  PRIMARY KEY (resource_id, enrollment_id)
```

The resource-access unit is now the **enrollment**, not the
individual session. A student who is enrolled in a course can
access the course's `enrolled`-visibility resources.

#### `_bookings_legacy` (renamed from `bookings`)

The old `bookings` table is renamed to `_bookings_legacy` by the
Sprint B2 migration. RLS is **off** on this table. New code must
not read it. It will be dropped in a later cleanup migration.

## 4. Enums

- `user_role`                 : `student | admin | super_admin`
- `booking_status`            : `scheduled | confirmed | completed | cancelled | no_show | rescheduled`  *(no more `pending_payment` — see §4.1)*
- `payment_status`            : `pending | succeeded | failed | refunded | partially_refunded`
- `payment_provider`          : `stripe | other`
- `enrollment_status`         : `pending_payment | active | completed | cancelled | refunded`  *(new)*
- `module_progress_status`    : `not_started | in_progress | completed`  *(new)*

### 4.1 Why no `pending_payment` on `module_bookings`

Payment happens at the **course** level (`enrollments`), not at
the **module** level. A `module_bookings` row is therefore never
"awaiting payment". The lifecycle is:

- `scheduled` — the Calendly `invitee.created` webhook fired, the
  row exists, but the Zoom meeting has not been created yet.
- `confirmed` — the Zoom meeting has been created and the
  `meeting_links` row is in place.
- `completed` — the Zoom `meeting.ended` webhook fired (or a
  tutor / admin confirmed the session).
- `cancelled` — the student (or admin) cancelled the booking.
- `no_show` — the student did not attend and the tutor / admin
  marked the session as such.
- `rescheduled` — the student picked a new slot; the row is kept
  for history, a new `module_bookings` row is created.

## 6. Row-Level Security

### 6.1 Pre-existing RLS (unchanged)

- `profiles` — owner-or-admin read; owner self-insert on signup; only
  super admin may delete.
- `tutors` — public read of `is_published = true`; admin write.
- `courses` — public read of `is_published = true`; admin write.
- `course_tutors` — public read; admin write.
- `resources` — public or `enrolled` (joined via `resource_grants`) or
  `private` (only the uploader).
- `notifications` — owner read, owner update, admin/owner insert.
- `audit_logs` — admin read only; inserts come from triggers.

### 6.2 New RLS (Sprint B2)

#### `modules`

- `SELECT` : `is_published = true`  **OR**  `is_admin()`
- `INSERT` / `UPDATE` / `DELETE` : `is_admin()` OR `is_super_admin()`

#### `enrollments`

- `SELECT` : `student_id = auth.uid()`  **OR**  tutor of one of the
  course's modules  **OR**  `is_admin()`
- `INSERT` : `deny`  (only the Stripe webhook / service role can write;
  the application goes through `POST /api/enrollments` which uses
  the server client + a SECURITY DEFINER RPC)
- `UPDATE` : `deny`  (the only writer is n8n with the service-role key
  on the `checkout.session.completed` event)
- `DELETE` : `deny`

#### `module_progress`

- `SELECT` : same as the corresponding `enrollments` row
- `INSERT` / `UPDATE` : service_role (via n8n) OR `is_admin()`

#### `module_bookings`

- `SELECT` : `student_id = auth.uid()`  **OR**  tutor of the module's
  course  **OR**  `is_admin()`
- `INSERT` : `deny`  (the only writer is n8n with the service-role key
  on the Calendly `invitee.created` event). The application goes
  through `POST /api/module-bookings` which uses a SECURITY DEFINER
  RPC and verifies that the student has an `active` enrollment
  for the module's course.
- `UPDATE` : service_role  **OR**
  (`student_id = auth.uid()` AND status IN
  ('scheduled', 'confirmed', 'cancelled'))
- `DELETE` : `deny`

#### `payments` (changed)

- `SELECT` : student can read payments of their own enrollments
- `UPDATE` : `deny` (only the Stripe webhook / n8n can write)

#### `meeting_links` (changed)

- `SELECT` : same visibility as the parent `module_bookings` row
- `INSERT` / `UPDATE` / `DELETE` : service_role only

#### `resource_grants` (changed)

- `SELECT` : student can read grants for their own enrollments
- `INSERT` / `UPDATE` / `DELETE` : service_role only

#### `_bookings_legacy`

- **RLS off.** No RLS policies are created on this table. The
  application must not query it.

## 8. Migrations

Numbered and idempotent. Sprint B2 adds **one** new migration and
**modifies** the existing RLS migration to cover the new tables:

```
supabase/migrations/
├── 20260707000001_extensions_and_helpers.sql
├── 20260707000002_profiles_and_roles.sql
├── 20260707000003_tutors_courses.sql
├── 20260707000004_bookings_payments.sql      -- renamed bookings → _bookings_legacy
├── 20260707000005_resources_notifications_audit.sql
├── 20260707000006_rls_policies.sql           -- augmented with new RLS
├── 20260707000007_storage_buckets.sql
├── 20260707000008_subscriptions_billing.sql
└── 20260709000001_modules_enrollments.sql    -- NEW (Sprint B2)
```

The new migration creates `modules`, `enrollments`,
`module_progress`, `module_bookings`; renames `bookings` to
`_bookings_legacy`; adds new columns to `payments`,
`meeting_links`, `resource_grants`; creates new enums; creates
new indexes; creates new RLS policies; creates the new
completion triggers.

Run them with the Supabase CLI:

```bash
supabase db push           # apply to linked project
supabase db reset          # drop + migrate + seed (dev only)
```

## 11. New triggers (Sprint B2)

- `enrollments_completion_trigger` — when every row in
  `module_progress` for an `enrollment_id` is `completed`, set
  `enrollments.status = 'completed'` and `completed_at = now()`.
- `module_bookings_completion_trigger` — when a `module_bookings`
  row transitions to `completed`, set the matching
  `module_progress` row to `completed`, `completed_at = now()`.
- The existing `fn_audit_changes` is extended to
  `enrollments`, `module_bookings`, `modules`.
- The existing `set_updated_at` trigger is added to
  `enrollments`, `modules`, `module_progress`,
  `module_bookings`.
